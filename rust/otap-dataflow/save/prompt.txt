Now, the way I understand, we have the pipeline controller iterating
 over the frames as it unwinds the context, and for the nodes with
 metrics it will output producer and consumer metrics. My impression
 is that we can compute one timestamp per routing of the ack/nack,
 that the one timestamp would be used for each metric recorded on the
 way back to the first ack/nack subscriber, and then a new timestamp
 is captured when it subsequently routes the next ack/nack.

The metric Coutners and Mmsc (histogram type) instrument are currently
located in a per-channel metric handle. Presently we have TODOs in the
place where metrics should be counted. Lets move the consumed_ and
produced_ metric Counter/Mmscs into new metric handles that will be
owned exclusively by the pipeline controller so there are no Arcs, the
pipline controller owning all ack/nack metric accoutning responsibility.
